Fleet Prompt — Orchestration Architecture

How should n8n, AURA CLI, GitHub, and agent skills be coordinated? Pick the model that resonates.

Option A

🔀 n8n-Centric Hub

n8n is the brain. Every analysis, PR action, and merge decision flows through n8n workflows. AURA CLI and GitHub CLI are called as HTTP tools.

GitHub Issues/PRs ↓ webhook n8n Dispatcher ├─ Skill Runner → AURA Skills API ├─ Code Eval → AURA CLI ├─ PR Creator → gh CLI (HTTP) └─ Merger → git API
✓ Visual workflow editor ✓ Built-in retry/fan-out ✓ Easy to extend
✗ n8n is single point of failure ✗ Hard to version-control logic
★ Recommended

🤖 AURA Fleet Commander

AURA CLI drives a fleet of specialized sub-agents. n8n handles async coordination + webhooks. GitHub is the shared state store (issues = goals, PRs = results).

GitHub Issues ←→ Git ↓ poll / webhook AURA FleetCommander ├─ Analyst Agent (skills) ├─ Planner Agent (decompose) ├─ Coder Agent (implement) ├─ Reviewer Agent (critique) └─ Merger Agent (gh pr merge) ↕ trace/feedback n8n Observability
✓ Code-first (versionable) ✓ n8n for visibility only ✓ Fits AURA loop
✗ More Python to build ✗ Requires AURA running
Option C

🌐 Peer Fleet + GitHub as Bus

GitHub Issues are the message bus. Each agent polls assigned labels, acts autonomously, and communicates via PR comments and issue updates. No central coordinator.

GitHub Issues (labeled) analyst:open → Analyst Agent plan:open → Planner Agent code:open → Coder Agent review:open → Reviewer Agent merge:approved → Merger Agent ↕ comments/labels n8n watches for stalls AURA CLI = agent runtime
✓ Fully decentralized ✓ Survives any single failure ✓ Audit trail in GitHub
✗ Slower (polling) ✗ Complex label state machine ✗ Harder to debug
