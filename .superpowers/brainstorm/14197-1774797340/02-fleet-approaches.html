Fleet Prompt — 3 Implementation Approaches

Trigger: GitHub Issue → Autopilot → Retry×3 → Blocker comment. Which architecture fits best?

Approach 1

📏 Linear Pipeline

One n8n workflow. Issue fires it, nodes run sequentially. Each node calls AURA Skills API and GitHub CLI. Simple but no parallelism.

Issue Trigger ↓ [Analyze] → AURA skills ↓ [Plan] → LLM prompt ↓ [Code] → AURA coder ↓ [Test] → pytest/lint ↓ [Push PR] → gh CLI ↓ [Merge] → gh CLI ↓ (on fail ×3) [Blocker comment]
✓ Easy to build & debug
✓ One workflow to maintain
✓ Sequential = predictable
✗ No goal-type specialization
✗ Slow (all steps serial)
✗ One prompt fits all issues
⚡ Build complexity: Low · 1 workflow
★ Recommended

🗂️ Type-Router + Sub-Flows

Master dispatcher classifies each issue (bug / feature / refactor / security / docs) and routes to a specialized sub-workflow with a tuned prompt chain.

Issue Trigger ↓ [Classifier] ← LLM prompt ├─ bug → Bug-Fix Flow ├─ feature → Feature Flow ├─ refactor → Refactor Flow ├─ security → Security Flow └─ docs → Docs Flow ↓ (all flows) [AURA Skills fan-out] [Coder + Tests] [Push PR + Merge] ↓ (on fail ×3) [Blocker comment]
✓ Specialized prompts per type
✓ Parallel skills analysis
✓ Reusable sub-flows
✓ Easy to add new types
✗ ~5 workflows to build
✗ Classifier can mis-route
⚡ Build complexity: Medium · 6 workflows
Approach 3

🔌 n8n → AURA Handoff

n8n handles only GitHub I/O (issue in, PR out). AURA's existing LoopOrchestrator does all planning, coding, and verification internally. Minimal new n8n code.

Issue Trigger ↓ [Enqueue Goal] → AURA API ↓ (POST /run) AURA runs full loop: ingest→plan→code →sandbox→apply →verify→reflect ↓ [Poll for result] ↓ [Push PR + Merge] → gh CLI ↓ (on fail ×3) [Blocker comment]
✓ Reuses all existing AURA logic
✓ Minimal n8n code (2 workflows)
✓ AURA loop already battle-tested
✗ AURA must be running 24/7
✗ Less visible inside n8n
✗ Harder to tune per issue type
⚡ Build complexity: Low · 2 workflows
